Jour 1 · Fondamentaux · Technologies · Architecture

De l'émergence des premiers systèmes distribués à l'ère du Lakehouse et de l'IA générative — 35 ans qui ont redéfini le traitement de la donnée.

1990 – 1999
L'ère relationnelle et ses limites
Les SGBD relationnels (Oracle, DB2, SQL Server) dominent. Tout tient sur un seul serveur. Le terme "Data Warehouse" émerge avec Inmon (1992). Les volumes explosent avec le Web commercial mais les architectures ne suivent pas. Les DBA parlent de "crises de scalabilité" : la verticalisation (ajouter de la RAM, du CPU) a ses limites physiques et économiques.
SQL OLAP Data Warehouse Scalabilité verticale
2003 – 2006
Les papers fondateurs de Google
Google publie trois papers qui vont tout changer : GFS (Google File System, 2003) — stockage distribué sur machines ordinaires, MapReduce (2004) — paradigme de calcul distribué en deux phases (map + reduce), et BigTable (2006) — base de données colonnaire distribuée. Ces papers posent les fondations conceptuelles du Big Data tel qu'on le connaît aujourd'hui.
GFS MapReduce BigTable Distribué
2006 – 2010
Naissance de Hadoop — l'open source s'empare du Big Data
Doug Cutting et Mike Cafarella créent Hadoop en s'inspirant des papers Google. Yahoo! est le premier à le déployer à grande échelle (10 000 nœuds en 2008). Apache Hadoop devient la référence open source : HDFS pour le stockage, MapReduce pour le calcul. L'écosystème s'enrichit rapidement : Hive (SQL sur Hadoop), Pig, HBase (clone open source de BigTable), ZooKeeper.
Apache Hadoop HDFS Hive HBase Open Source
2009 – 2013
Explosion NoSQL — "Not Only SQL"
Le modèle relationnel montre ses limites pour certains workloads : MongoDB (documents JSON, 2009), Cassandra (colonnaire distribué, partagé par Facebook), Redis (clé-valeur en mémoire), Neo4j (graphes). Le théorème CAP (Brewer, 2000) entre dans le vocabulaire courant : on ne peut garantir simultanément Cohérence, Disponibilité et Tolérance aux partitions. Le terme "Big Data" est popularisé et le Gartner le place dans son Hype Cycle.
MongoDB Cassandra Redis CAP Theorem Neo4j
2012 – 2016
Spark, Kafka et le temps réel — MapReduce n'est plus suffisant
Apache Spark (Matei Zaharia, Berkeley, 2009 → Apache 2013) révolutionne le traitement : calcul en mémoire 100× plus rapide que MapReduce, API Python/Scala/Java, ML intégré (MLlib). Apache Kafka (LinkedIn, 2011 → Apache 2012) invente le bus d'événements distribué. L'architecture Lambda (Nathan Marz) unifie batch et stream. L'ère du "real-time analytics" commence.
Apache Spark Apache Kafka Lambda Arch. Streaming MLlib
2015 – 2019
Cloud & Data Lake — la démocratisation
AWS S3, Google Cloud Storage et Azure Blob Storage deviennent les nouveaux HDFS. Le concept de Data Lake (stockage brut, schema-on-read) s'oppose au Data Warehouse (schema-on-write). Databricks popularise Delta Lake. Les services managés (EMR, Dataproc, HDInsight) font du Big Data accessible aux PME. Le coût du stockage s'effondre (1 To sur S3 ≈ 23$/mois). Snowflake lève 480M$ en 2018.
AWS S3 Data Lake Snowflake Delta Lake Serverless
2020 – 2022
Lakehouse — convergence Data Lake + Data Warehouse
Databricks introduit l'architecture Lakehouse : le meilleur des deux mondes — la flexibilité et le coût du Data Lake + les performances ACID et l'optimisation des requêtes du Data Warehouse. Formats ouverts : Delta Lake (Databricks), Apache Iceberg (Netflix), Apache Hudi (Uber). L'architecture Médaillon (Bronze/Silver/Gold) structure les pipelines de transformation. dbt s'impose pour la transformation SQL.
Lakehouse Apache Iceberg Médaillon dbt ACID
2023 – 2025
IA générative & Data Mesh — la nouvelle frontière
ChatGPT (Nov. 2022) change la perception du Big Data : les LLMs sont entraînés sur des données massives et redistribuent la valeur vers la qualité des données plutôt que le volume brut. Le Data Mesh (Zhamak Dehghani) décentralise la gouvernance : chaque domaine métier est owner de ses données. Les Vector Databases (Pinecone, Weaviate, pgvector) émergent pour la RAG. La Modern Data Stack se consolide autour de Fivetran/dbt/Snowflake/Looker.
LLM / GenAI Data Mesh Vector DB RAG MLOps

Architectures hybrides
Architecture Lambda
Proposée par Nathan Marz (créateur de Storm). Combine deux couches parallèles : une batch layer (Hadoop/Spark) qui recalcule les vues complètes sur l'historique, et une speed layer (Kafka/Storm) qui traite les données récentes en temps réel. Une serving layer (HBase, Cassandra) fusionne les deux pour les requêtes.
Tolérance aux pannes — l'historique batch corrige les erreurs stream
Adapté aux grands volumes historiques + temps réel
Deux codebase à maintenir (batch + stream)
Complexité opérationnelle élevée
Hadoop Apache Storm HBase
Architecture Kappa
Proposée par Jay Kreps (co-créateur de Kafka). Simplification radicale : une seule couche stream pour tout traiter. Le "batch" devient un stream replay : on rejoue l'historique depuis Kafka pour recalculer les vues. Principe : si le stream processing est assez puissant (Flink, Kafka Streams), pas besoin d'un batch séparé.
Un seul codebase — maintenance simplifiée
Cohérence garantie entre historique et temps réel
Kafka doit stocker toute la rétention nécessaire
Moins adapté aux calculs historiques très complexes
Apache Kafka Apache Flink Kafka Streams
→ Tendance 2024 : L'architecture Lakehouse (Databricks, Apache Iceberg) tend à remplacer Lambda en unifiant batch et stream dans une même couche de stockage ACID, réduisant encore la complexité.

Consistance (C) — Cohérence forte
Après une écriture, toute lecture renvoie la valeur la plus récente. Tous les nœuds du cluster sont synchronisés avant de répondre.

Exemple : Un virement bancaire — si je transfère 500€, mon solde doit être immédiatement correct sur tous les serveurs.

Coût : Nécessite des verrous distribués (2PC, Paxos, Raft) → latence plus élevée.

ACID Paxos Raft 2PC
Disponibilité (A) — Haute disponibilité
Le système répond toujours à chaque requête — même si la donnée n'est pas la plus récente. Aucun nœud sain ne refuse une requête.

Exemple : Netflix continue d'afficher vos recommandations même si un datacenter est en panne. La liste peut être légèrement obsolète, mais vous ne voyez pas d'erreur.

Coût : Accepter une éventuelle incohérence temporaire → modèle BASE (vs ACID).

BASE Eventual consistency Réplication async
Tolérance au partitionnement (P) — Partition tolerance
Le système continue de fonctionner même si des nœuds ne peuvent plus communiquer entre eux (coupure réseau, perte de paquets).

Exemple : Un cluster Cassandra en 3 datacenters continue de fonctionner si le lien réseau entre Europe et USA est coupé.

Réalité : Dans les systèmes distribués, P est quasi-obligatoire. Le vrai choix est donc entre C et A.

Split-brain Network partition Gossip protocol
